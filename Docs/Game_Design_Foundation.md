# Game Design Foundation

이 문서는 이 프로젝트가 어떤 게임을 목표로 하는지, 어떤 순서로 개발해야 하는지, 왜 현재 구조를 이렇게 잡는지 정리한다.

## 게임 한 줄 설명

2D 탑다운 실시간 액션 기반의 싱글플레이 Extraction RPG.

플레이어는 던전에 들어가 적과 싸우고, 전리품을 찾고, 살아서 탈출해야 보상을 확보한다.

## 핵심 루프

가장 중요한 플레이 흐름:

```text
던전 입장
→ 탐색
→ 전투
→ 적 처치
→ 시간 소모 루팅
→ 아이템 획득
→ 탈출구 찾기
→ 시간 소모 탈출
→ 마을 복귀
→ 판매/정산
→ 골드 획득
→ 재입장
```

이 루프가 안정되기 전에는 고급 시스템을 크게 추가하지 않는다.

## 현재 개발 단계

현재 단계:

```text
v0.1 핵심 루프 프로토타입
```

v0.1에서 반드시 필요한 것:

- 플레이어 이동
- 마우스 방향 조준
- 기본 공격
- 플레이어 체력
- 적 AI
- 적 체력
- 적 처치
- 아이템 드랍
- 시간 소모 루팅
- 아이템 인벤토리 이동
- 탈출구 상호작용
- 시간 소모 탈출
- 탈출 성공 처리
- 아이템 판매
- 골드 획득
- 던전 재입장

## 게임의 핵심 재미

이 게임의 핵심은 단순히 아이템을 많이 줍는 것이 아니다.

중요한 감정:

- 더 파밍하고 싶다.
- 하지만 오래 머무르면 위험하다.
- 좋은 아이템을 찾았으니 탈출하고 싶다.
- 탈출 중에도 방해받을 수 있다.
- 죽으면 이번 원정에서 얻은 것을 잃는다.

그래서 루팅과 탈출은 즉시 완료되지 않아야 한다.

## 왜 시간 소모 루팅이 중요한가

일반 RPG에서는 적을 죽이면 아이템이 바로 떨어지고 바로 줍는다. 하지만 Extraction RPG에서는 "가져갈 수 있느냐"가 재미의 핵심이다.

시간 소모 루팅을 넣으면 다음 선택이 생긴다.

- 지금 수색할 것인가?
- 적이 더 올 수 있는데 기다릴 것인가?
- 보물상자를 열 것인가, 탈출할 것인가?
- 맞으면 취소되는데 위험을 감수할 것인가?

이 구조는 나중에 다음 시스템과 연결된다.

- 소리 감지
- 적 증원
- 수색 속도 스탯
- 장비 무게
- 직업 특성
- 유전자/변이 특성
- 탈출 아이템
- 잠긴 상자

## 초기에는 만들지 않을 시스템

v0.1이 안정되기 전까지 미룰 것:

- 직업
- 유전자/변이
- 계승
- 세금/부채
- 보스
- 랜덤 던전 생성
- 복잡한 장비 강화
- 복잡한 퀘스트
- 대규모 스토리
- 창고 전체 시스템
- 복잡한 상점 구매 시스템

이유:

- 핵심 루프가 재미있는지 확인하기 전에 시스템을 많이 붙이면 문제 원인을 알기 어렵다.
- 고급 시스템은 대부분 아이템, 전투, 경제, 탈출 루프 위에 올라간다.
- 기반 루프가 약하면 고급 시스템도 재미를 만들기 어렵다.

## 데이터와 런타임 상태 분리

아이템 시스템은 다음 원칙을 따른다.

### ItemData

역할:

- 변하지 않는 기본 데이터
- 이름
- 아이콘
- 크기
- 타입
- 판매 가격
- 공격 보너스

주의:

- ScriptableObject를 플레이어 저장 데이터처럼 직접 수정하지 않는다.

### InventoryItem

역할:

- 화면에 표시되는 런타임 아이템 UI
- 어느 격자에 있는지
- 몇 칸을 차지하는지

### LootContainer

역할:

- 적 시체, 주머니, 상자 같은 대상이 가진 전리품 목록
- 플레이어가 가져가기 전까지는 플레이어 소유가 아니다.

### InventoryController

역할:

- 플레이어가 실제로 가진 인벤토리를 관리한다.
- 아이템 추가, 제거, UI 아이템 생성 흐름을 담당한다.

왜 이렇게 나누는가:

- 전리품과 플레이어 소유 아이템을 섞으면 사망 시 손실, 탈출 성공 보상, 상점 판매가 꼬인다.
- 나중에 창고, 안전 주머니, 장비 내구도, 희귀 옵션이 추가될 때 확장하기 쉽다.

## UI 설계 방향

현재 목표 UI:

- Tarkov식 하드코어 인벤토리 느낌
- 어두운 반투명 패널
- 격자 기반 아이템 배치
- 아이템 크기에 따른 공간 차지
- 플레이어 가방과 대상 컨테이너를 동시에 표시
- 툴팁 중심 정보 표시

중요 원칙:

- UI 전체를 통짜 이미지로 만들지 않는다.
- 레이아웃은 Unity `RectTransform`으로 구성한다.
- 패널과 슬롯은 `Image`로 조립한다.
- 텍스트는 `Text` 또는 나중에 `TextMeshPro`를 사용한다.
- 아이템 아이콘은 Sprite로 관리한다.

## 개발 원칙

- 모르면 모른다고 말한다.
- 기존 코드를 확인하고 수정한다.
- 핵심 루프를 먼저 완성한다.
- 한 번에 모든 시스템을 만들지 않는다.
- 런타임 데이터와 기본 데이터를 섞지 않는다.
- UI와 게임 로직을 강하게 묶지 않는다.
- Inspector에서 수정 가능한 값은 한글 Header/Tooltip을 붙인다.
- 변경 후에는 Null 가능성, 레이어, 프리팹 연결, 테스트 방법을 확인한다.

## 현재 프로젝트의 가장 중요한 방향

지금 가장 중요한 것은 예쁜 최종 UI나 거대한 장비 시스템이 아니다.

가장 먼저 완성해야 하는 것은 다음 감각이다.

```text
위험한 던전에서 전리품을 찾았다.
가져가려면 시간이 걸린다.
그동안 공격받으면 취소된다.
살아서 탈출해야 보상이 된다.
그 보상으로 다음 원정을 준비한다.
```

이 감각이 살아나면 이후 직업, 장비, 유전자, 세금, 계승 같은 시스템은 자연스럽게 얹을 수 있다.
