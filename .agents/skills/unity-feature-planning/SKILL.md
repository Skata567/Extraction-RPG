# Unity 기능 계획 스킬

이 스킬은 새 기능, 리팩토링, 시스템 확장을 시작하기 전에 사용한다.
목표는 큰 설계를 한 번에 확정하는 것이 아니라, 현재 단계에 맞는 작은 작업 단위로 쪼개는 것이다.

## 언제 사용할 것

- 새 기능을 구현하기 전
- 기존 시스템을 리팩토링하기 전
- 사용자가 아이디어를 제안했을 때
- 작업 범위가 씬, 프리팹, ScriptableObject, 여러 스크립트에 걸릴 때
- 이 기능이 v0.1에 필요한지 판단해야 할 때

## 먼저 답해야 할 질문

1. 이 기능이 v0.1 핵심 루프에 필요한가?
2. 지금 개발 단계에 맞는가?
3. 구현 난이도는 어느 정도인가?
4. 다른 시스템과 역할이 겹치지 않는가?
5. 나중으로 미뤄도 되는가?
6. MVP에 반드시 필요한가?
7. 구현한다면 어느 파일, 씬, 프리팹, ScriptableObject와 연결되는가?
8. 확장성을 위해 어떤 연결점만 남기면 충분한가?

## 설계 기준

- 하나의 기능은 가능한 한 하나의 PR 또는 작은 작업 단위로 나눈다.
- 기능 추가와 대규모 리팩토링은 분리한다.
- MonoBehaviour는 입력, 이동, 전투, 상호작용, UI처럼 책임별로 나눈다.
- 데이터가 여러 개 생길 가능성이 높으면 ScriptableObject 후보로 본다.
- 단순 테스트용 값은 처음부터 과하게 ScriptableObject로 만들지 않고 Inspector 설정으로 시작해도 된다.
- UI는 게임 로직을 직접 소유하지 않고 표시만 담당한다.
- 나중에 확장될 부분은 인터페이스나 이벤트를 무리하게 도입하기보다 메서드/데이터 경계를 분명히 두는 정도로 시작한다.

## 위험 점검

계획 단계에서 아래를 반드시 확인한다.

- 기존 씬이나 프리팹을 크게 바꾸는가?
- Inspector 연결이 필요한 필드가 생기는가?
- 태그/레이어/Sorting Layer 설정이 필요한가?
- Manager가 씬 전환 시 중복 생성될 가능성이 있는가?
- ScriptableObject 기본 데이터와 런타임 상태가 섞일 가능성이 있는가?
- 기능이 현재 핵심 루프보다 고급 시스템에 가까운가?
