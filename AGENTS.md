# Unity 2D Extraction RPG 작업 지침

이 프로젝트는 Unity 2D 기반 싱글플레이 Extraction RPG를 단계적으로 개발하는 프로젝트다.
목표는 완성형 게임을 한 번에 만드는 것이 아니라, v0.1 핵심 루프를 먼저 안정적으로 구현하고 이후 시스템을 확장하는 것이다.

## 현재 우선순위

- 현재 단계는 `v0.1 핵심 루프 프로토타입`이다.
- 던전 입장, 전투, 파밍, 탈출, 판매, 골드 획득, 재입장이 자연스럽게 이어지는 것을 최우선으로 한다.
- 핵심 루프가 완성되기 전에는 직업, 유전자/변이, 계승, 세금/부채, 보스, 랜덤 던전, 복잡한 장비 강화 같은 고급 시스템을 구현하지 않는다.

## 작업 전 확인

- 코드를 수정하기 전에 Unity 버전, 패키지, 씬, 프리팹, 기존 스크립트 구조를 먼저 확인한다.
- 존재하지 않는 Unity API, 패키지, 클래스, 메서드를 있다고 가정하지 않는다.
- 기존 코드 일부만 보고 전체 구조를 단정하지 않는다.
- 필요한 파일, 오류 메시지, 씬 구조, Inspector 연결 상태가 부족하면 확인이 필요하다고 말한다.
- 진행 가능한 합리적 가정은 반드시 `가정`이라고 표시한 뒤 진행한다.

## 개발 원칙

- 큰 변경보다 작은 변경 단위를 우선한다.
- 기능 추가와 대규모 리팩토링은 가능하면 분리한다.
- 기존 네이밍, 폴더 구조, 아키텍처가 있다면 가능한 한 맞춘다.
- 하나의 거대한 스크립트에 모든 기능을 몰아넣지 않는다.
- 플레이어, 전투, 적, 아이템, 인벤토리, 던전, 경제, UI의 책임을 분리한다.
- 하드코딩과 매직 넘버를 줄이고, Inspector 설정값이나 ScriptableObject로 분리한다.
- 아직 필요 없는 구조를 과하게 미리 만들지 않는다.
- 목표는 `현재 필요한 최소 구현 + 나중에 확장 가능한 연결점`이다.

## Unity / C# 작성 규칙

- `MonoBehaviour`는 역할별로 작게 나눈다.
- Inspector에 노출되는 필드는 기본적으로 `private` + `[SerializeField]`를 사용한다.
- Inspector 설정에는 `[Header]`, `[Tooltip]`을 붙이고, 사용자가 의미를 바로 알 수 있도록 한글로 작성한다.
- `public` 필드는 외부에서 반드시 접근해야 할 때만 사용한다.
- 공통 런타임 로직은 가능하면 순수 C# 클래스로 분리해 테스트하기 쉽게 만든다.
- `Health`처럼 플레이어와 적이 공유할 수 있는 기능은 공통 컴포넌트로 설계한다.
- `ItemData`, `WeaponData`, `EnemyData` 같은 ScriptableObject는 변하지 않는 기본 데이터로 사용한다.
- 플레이어가 소유한 아이템 상태, 원정 중 획득한 아이템, 골드 같은 런타임 데이터는 ScriptableObject 원본에 직접 저장하지 않는다.
- 빈 `catch` 블록이나 오류를 조용히 무시하는 코드를 남기지 않는다.
- Null, 빈 배열, 잘못된 입력, Inspector 참조 누락 가능성을 고려한다.

## 주석 규칙

- 코드에는 필요한 곳에 한글 주석을 단다.
- 주석은 단순히 무엇을 하는지가 아니라 왜 이렇게 설계했는지를 설명한다.
- 너무 뻔한 주석은 피한다.
- 임시 코드, 우회 코드, 나중에 고칠 부분은 `TODO` 또는 `FIXME`로 표시한다.

좋은 주석 예시:

```csharp
// 던전에서 탈출하기 전까지 얻은 아이템은 임시 인벤토리에 보관한다.
// 나중에 장비, 직업, 유전자 보정치를 추가할 수 있도록 데미지 계산을 별도 메서드로 분리한다.
// Scene 뷰에서 적의 감지 범위를 확인하기 위해 Gizmo를 그린다.
```

## 테스트와 검증

- 코드 작성 후 Unity 컴파일 오류를 확인한다.
- 가능한 경우 EditMode 테스트 또는 PlayMode 테스트를 작성한다.
- 순수 로직인 인벤토리, 판매 계산, 골드 증감, 원정 정산은 테스트 우선 적용을 고려한다.
- 이동, 전투, 물리, 씬 전환, UI 상호작용은 수동 플레이 테스트 절차를 남긴다.
- 변경 후 다음 항목을 반드시 점검한다.
  - `NullReferenceException` 가능성
  - Inspector 참조 누락
  - 프리팹 연결 누락
  - 태그/레이어 설정 누락
  - 씬 전환 시 Manager 중복 생성
  - ScriptableObject 기본 데이터와 런타임 데이터 혼동
  - 기존 기능 회귀

## 보안과 위험 작업

- API 키, 토큰, 비밀번호, 개인정보를 코드나 씬에 하드코딩하지 않는다.
- 패키지 추가, ProjectSettings 변경, 대량 파일 이동/삭제, Git push, PR 생성은 먼저 범위와 위험을 알린다.
- 외부 패키지를 추가하기 전에 기존 Unity 기능이나 현재 의존성으로 해결 가능한지 먼저 검토한다.
- 파일 삭제, 데이터 손실 가능 작업, 원격 저장소 변경은 신중하게 처리한다.

## 로컬 스킬

이 프로젝트에는 아래 로컬 스킬을 둔다. 관련 작업을 할 때는 해당 스킬의 기준을 우선 참고한다.

  - Unity C# 코드 작성 또는 수정 후 리뷰에 사용한다.
  - MonoBehaviour 책임, Inspector 참조, ScriptableObject 오용, Physics2D 설정, Null 가능성을 점검한다.
- `.agents/skills/unity-verification-loop/SKILL.md`
  - 기능 구현, 버그 수정, PR 준비 전 검증에 사용한다.
  - Unity 컴파일, 테스트, 씬/프리팹/Inspector 연결, 핵심 루프 수동 테스트를 확인한다.

## PR 기준

- 하나의 PR은 하나의 명확한 목표만 가진다.
- PR 전에는 변경 파일, 핵심 로직, 테스트 방법, 현재 한계를 정리한다.
- 커밋이나 PR 설명에는 사용자가 확인할 수 있는 테스트 절차를 포함한다.
- 검증하지 못한 항목은 검증했다고 말하지 않는다.
